# 账号系统 Backlog

## Summary

账号系统不再被设计为“替代同步码”的单一同步方案，而是产品的身份、空间管理、全局偏好和会员权益层。

新的长期模型：

```text
Account
  -> Profile
  -> Global Preferences
  -> Billing / Entitlements
  -> Home Spaces
       -> Home Document
       -> Sync Codes
       -> Devices
```

在这个模型下：

- 账号登录是上级身份层。
- 同步码是某个首页空间的访问凭证。
- 一个账号可以挂载一个或多个首页空间。
- 每个首页空间可以有一个或多个同步码。
- 登录账号后，用户可以选择和切换账号下的首页空间。
- 账号负责同步展示风格、系统设置、语言、字体、会员状态等全局信息。
- 同步码继续保留低摩擦跨设备同步价值，不和账号登录互斥。
- 登录用户的长期默认体验应升级为账号托管首页空间：账号托管加密凭证，用户不直接管理同步码，空白设备登录后可以直接恢复空间。

## Product Position

账号系统承担三类长期价值：

- 身份价值：让用户在不同设备上识别自己，管理自己的首页空间、同步码和设备。
- 配置价值：同步语言、主题、字体、默认搜索引擎、组件默认行为等全局偏好。
- 商业价值：作为 VIP 付费、空间数量、组件权限、历史版本、AI 额度等权益认证入口。

## Deployment And Domain Roadmap

账号系统、首页空间和域名体系后续应统一到一个稳定的主域名和空间路由模型下：

- 短期：
  - 继续使用 GitHub Pages。
  - 绑定自购主域名，逐步摆脱 `/PersonalHomepge/` 项目路径。
  - Supabase Auth 的 `Site URL` 和 `Redirect URLs` 切换到统一主域名。
- 中期：
  - 评估迁移到 Vercel 或 Cloudflare Pages。
  - 如果未来重点是 Next.js DX、预览环境和平台化部署，优先评估 Vercel。
  - 如果未来重点是 DNS、WAF、限流、Turnstile、R2 和 SaaS 域名托管，优先评估 Cloudflare。
- 长期：
  - VIP 用户支持自定义域名。
  - 自定义域名应绑定到某个 `home_space`，而不是直接绑定账号。
  - 编辑和账号管理仍通过主站后台完成；自定义域名默认服务于首页展示和访问，不直接暴露后台编辑入口。

长期模型补充：

```text
Account
  -> Home Spaces
       -> Sync Codes
       -> Custom Domains
       -> Public/Private Access Mode
```

域名系统应服务于 `home_spaces`，因为未来一个账号下可以有多个首页空间，而不是只有一个首页。

## Phase 1 Extraction

Phase 1 只抽取账号系统中能支撑 MVP 的部分，不引入付费、不做团队、不做复杂设备控制。

Phase 1.5 应覆盖：

- 账号登录：
  - 邮箱 Magic Link。
  - 登录状态展示。
  - 登出。
  - 刷新后保持 session。
- 账号全局设置骨架：
  - 先保存最小偏好字段，例如 `locale`、`defaultSpaceId`、`themePreference`。
  - 不要求立刻做完整主题和字体 UI。
- 账号首页空间索引：
  - 一个账号可以拥有多个 `home_spaces`。
  - 每个 `home_space` 先关联一个现有 `sync_spaces`。
  - 首页内容本身继续沿用现有同步码密文同步机制。
- 同步码认领：
  - 登录后可以把当前本机绑定的同步码挂到账户下。
  - 认领后，该同步码仍可继续被未登录设备使用。
  - 账号只是管理和索引它，不改变同步码的低摩擦能力。
- 首页空间切换：
  - 设置页显示账号下已认领的首页空间。
  - 用户可选择某个空间并把其同步码绑定到当前浏览器。
  - 切换空间必须提示会拉取该空间首页并覆盖当前本地显示。
- 安全边界：
  - 账号表启用 RLS。
  - 用户只能查看和管理自己认领的空间。
  - 同步码 secret 不应以明文长期展示在账号空间列表中。

Phase 1 暂不做：

- Stripe 或会员支付。
- 团队共享。
- 多同步码对应同一空间的完整凭证管理。
- 账号托管凭证和空白设备直接恢复空间内容。
- 公开主页。
- 高级组件权益。
- 历史版本回滚。
- 图片存储和高级主题上传。

## Updated Phase 1.5 Breakdown

### Phase 1.5.0：账号模型与数据库计划

- 明确账号是上级身份层，同步码是首页空间凭证。
- 设计 `profiles`、`account_preferences`、`home_spaces` 的最小表结构。
- 决定 `home_spaces` 如何关联当前 `sync_spaces`。
- 写清楚 RLS policy 和验收 SQL。

### Phase 1.5.1：Supabase Auth 基础登录

- 调整 Supabase browser client：
  - 开启 session 持久化。
  - 开启 token 自动刷新。
  - 开启 URL session 检测。
- 设置页账号区支持：
  - 输入邮箱。
  - 发送 Magic Link。
  - 显示当前账号。
  - 登出。
- 不写首页空间数据。

### Phase 1.5.2：Resend Custom SMTP

- 配置 Resend 作为 Supabase Auth Custom SMTP。
- 完成发信域名 DNS 验证，包括 SPF、DKIM 和必要的 DMARC。
- 在 Supabase Auth SMTP 设置中配置 Resend SMTP 参数。
- 调整 Supabase Auth 邮件发送 rate limits，支撑本地和线上连续测试。
- 回归 Magic Link 注册/登录、登出后重新登录、GitHub Pages 回调。
- 本阶段不写账号业务表，不改同步码协议，不实现空间认领。

### Phase 1.5.3：账号资料与全局设置骨架

- 创建最小账号资料和偏好表。
- 登录后读取或初始化账号偏好。
- 先展示语言、主题、默认空间等占位状态。
- 不在本阶段实现完整主题/字体配置 UI。

### Phase 1.5.4：账号首页空间索引与同步码认领

- 新增账号下的 `home_spaces`。
- 登录后可以把当前本机绑定的同步码认领到账户。
- 认领时用户需要填写或确认空间名称。
- 已认领空间在设置页账号区域中展示。

### Phase 1.5.5：首页空间切换

- 设置页展示账号下空间列表。
- 点击某个空间可切换当前浏览器绑定的同步码。
- 切换前二次确认。
- 切换后执行现有同步码拉取流程。
- 当前空间状态写入本地缓存和账号偏好。

### Phase 1.5.6：安全回归与线上验证

- 验证未登录用户不能读取账号表。
- 验证账号 A 不能读取账号 B 的空间列表。
- 验证登录前后同步码创建、绑定、上传、拉取、废弃仍正常。
- 验证 GitHub Pages 登录回调。
- 验证空间切换不会静默覆盖本地数据。

## Risk Notes

- 最大风险从“账号同步覆盖本地首页”变为“空间切换覆盖当前本地首页”。所有空间切换和拉取都必须二次确认。
- RLS 配置错误可能导致账号空间越权访问，必须单独验收。
- 同步码认领不能泄露同步码 secret；账号空间列表默认只展示空间元数据。
- 账号托管凭证允许空白设备恢复空间内容，安全边界从“持有同步码”转为“账号身份 + RLS”，需要单独验收跨账号隔离和前端泄露风险。
- Supabase Auth session 配置变化可能影响现有同步码 RPC，需要完整回归。
- GitHub Pages 是静态部署，需要提前验证 Auth redirect URL、base path 和 callback 行为。
- 自购主域名上线后，`basePath`、Supabase Auth URL 配置和本地缓存域隔离都会变化，需要单独回归。
- VIP 自定义域名不是简单 DNS 功能，而是 `home_space -> hostname` 的绑定系统，需要域名所有权验证、证书管理和空间路由。

## Recommended Position

Phase 1.5 已完成当前同步码空间模型的账号化收口：账号登录、空间索引、同步码认领、空间切换和安全回归均已落地。Phase 1.6 开始引入账号托管同步，把登录用户的默认体验从“手动同步码”升级为“账号管理首页空间”；同步码保留为匿名、恢复和高级入口。
