# 0-3 个月 MVP 产品设计与实施路线

## Summary

把当前静态首页渐进升级为可公测 MVP：先支持本地编辑，再用“同步码”实现跨设备同步，然后接入正式账号系统。0-3 个月终点是：用户不登录也能在当前浏览器编辑首页；使用同步码可跨设备恢复和同步；登录后账号可以管理多个同步码首页空间，并同步语言、展示风格、系统设置等全局偏好；页面仍保持极简、高频、快速打开的浏览器首页体验。

## Key Product Decisions

- 编辑体验从“查看/编辑切换”演进为“首页直编 + 独立设置页”：日常首页保持干净，分组和网站通过浮层菜单编辑，后台设置集中在 `/edit`。
- MVP 在未登录和单同步码模式下仍是单首页；进入账号阶段后，账号可以管理多个同步码首页空间，但每次浏览器只激活一个当前空间。
- 数据先采用“首页文档 JSON”模型，而不是一开始拆成复杂多表，降低 0-3 个月成本。
- 组件系统预留扩展结构，0-3 个月只做最小可用版本：Todo List、月历/万年历、基础组件布局管理和模板默认组件。
- 技术路线允许迁移：本地编辑先可基于当前页面快速验证，进入同步和登录阶段时迁移到 Next.js + Supabase。
- 账号登录不替代同步码，而是作为上级身份层管理同步码、首页空间、全局偏好和未来会员权益。
- AI 能力是 Phase 2 的核心方向之一，具体功能细节待定；方向上应服务于首页整理、搜索、清理、总结和工作流启动，而不是只做装饰性聊天入口。
- 浏览器收藏/标签导入是独立的大需求集，暂不放在 Phase 1.6 尾部实现。Phase 1.9 拆成两部分推进：先做前端页面布局和 UI/UX 优化，解决设置页、首页空间创建、图片设置面板和网站收集区编辑入口的低成本使用问题；再进入浏览器收藏/标签导入需求集，完整设计普通网页、用户主动导入文件、粘贴 URL 和浏览器扩展之间的权限边界。

Phase 1.9 拆成两个连续部分：

- Phase 1.9A 前端页面布局和 UI/UX 优化：收口设置页信息架构、首页空间创建流程、Banner/背景设置面板布局、网站收集区编辑入口，以及移动端/键盘可访问性。
- Phase 1.9B 浏览器收藏/标签导入需求集：在 UI/UX 优化完成后，再单独设计导入权限边界、大批量处理、预览确认和回滚策略。

Phase 1.9A 的关键交互原则：

- 桌面细指针设备可使用 hover/focus reveal，减少默认视觉负担。
- 触屏设备没有 hover，新增网站/新增分组等关键入口不能完全隐藏；应使用 `(hover: none)` 或 `(pointer: coarse)` 等输入能力判断，让触屏端保持可见的圆形按钮。
- 键盘用户必须能通过 `focus-within` 或可聚焦按钮发现入口。
- 不把长按、滑动露出等手势作为唯一入口；触屏按钮目标尺寸至少按 44px 级别设计。

Phase 1.9B 已知边界：

- 普通网页不能直接读取浏览器收藏夹或当前打开标签页。
- 用户收藏可能达到上千个，需要先设计大批量导入、清洗、去重、分组、抽样预览、性能和误导入回滚策略。
- 是否采用书签 HTML、粘贴 URL 列表、浏览器扩展或多种方案组合，留到 Phase 1.9B 需求设计阶段决定。

### 第 1 阶段：本地可编辑首页，约 1-2 周

- 在当前首页上加入编辑模式：
  - 新增/重命名/删除分组。
  - 新增/编辑/删除网站。
  - 支持调整分组和网站顺序，优先用上移/下移按钮，后续再加拖拽。
  - 删除操作必须二次确认。
- 数据保存到浏览器本地：
  - 使用 `localStorage` 或 IndexedDB 保存完整首页文档。
  - 未登录、未同步码状态下，所有编辑只影响当前设备当前浏览器。
  - 提供“导出 JSON”和“导入 JSON”，作为同步码前的兜底迁移方式。
- 保留现有能力：
  - DuckDuckGo 搜索。
  - 网站 favicon。
  - 极简浅色仪表盘布局。

### 第 2 阶段：统一数据结构与 Next.js 迁移，约 2-4 周

- 建立统一首页文档 schema：
  - `version`
  - `updatedAt`
  - `groups`
  - `sites`
  - `widgets`
  - `theme`
  - `syncMeta`
- 将首页迁移为 Next.js App：
  - 首页渲染、编辑器、搜索、favicon 逻辑组件化。
  - 本地存储继续可用，作为未登录模式和离线缓存。
  - 从旧 `homepage.html` 的默认链接生成初始模板。
- 保持一个明确入口：
  - `/` 为用户首页。
  - 编辑模式通过页面内按钮进入，不单独暴露复杂后台。

### 第 3 阶段：同步码跨设备同步，约 4-7 周

- 用户可创建同步空间：
  - 系统生成高强度随机同步码。
  - 同步码包含 `space_id` 和 `secret`，只有拿到同步码的人能读取/写入该配置。
- 云端保存策略：
  - 使用 Supabase 存储加密后的首页文档。
  - 服务端只保存密文和 revision，不保存明文内容。
  - 写入通过服务端 API 或 Supabase Edge Function 校验同步 secret，不能让前端直接任意改数据库。
- 同步体验：
  - 当前设备编辑后立即更新本地缓存，并 debounce 上传云端。
  - 新设备输入同步码后拉取配置，保存到本地缓存。
  - 多设备冲突时使用 revision 检测；MVP 不做自动合并，提示用户选择“使用本地版本”或“使用云端版本”。
- 安全边界：
  - 拿到普通页面网址不能编辑。
  - 拿到同步码等同拥有该首页编辑权。
  - 重新生成同步码属于后续 P4/未定义候选；MVP 先保留创建、绑定和废弃当前同步码。

### 第 4 阶段：前端展示页与 UI 交互优化，约 7-9 周

- 优化编辑体验：
  - 日常首页不再显示顶部状态 pill 和文字“编辑”按钮。
  - 编辑入口改为右侧状态展示区中 `Local` 旁边的齿轮图标。
  - 点击齿轮进入新的设置页，集中管理账号占位、同步码、配置文件和通用设置占位。
  - 分组和网站的新增、修改、删除入口保留在首页浮层菜单中。
  - 日常首页不在分组末尾展示“新增网站”按钮，避免首页管理控件过多。
  - 删除操作继续二次确认。
- 优化展示层级：
  - 增加固定“未分组”区域，放在所有分组最上方，横向布满网站收集区。
  - 日常首页的“未分组”只展示网站磁贴，不显示标题、管理按钮或外层卡片。
  - 将同步状态合并到右侧状态展示区，不再在顶部重复展示。
  - 日常首页保持轻量调整，不大幅重排。
  - 设置页保持极简后台管理风格，只放账号、同步码、配置文件和通用设置模块。
  - 桌面和移动端都要避免控件重叠、文字撑破和误触外链。
- 保留现有能力：
  - DuckDuckGo 搜索。
  - 拖拽排序。
  - 同步码自动同步。
  - 导入 JSON、导出 JSON、清空内容并恢复默认。
  - 恢复默认前自动保存最近一次本地备份，可恢复上一次重置前页面。
- 预留后续能力：
  - 设置页展示登录界面占位，但不实现登录功能。
- 本阶段边界：
  - 不做 Supabase Auth。
  - 不新增账号首页表。
  - 不改同步码协议。

### 第 5 阶段：账号登录与首页空间管理，约 9-11 周

- 接入 Supabase Auth：
  - MVP 首选邮箱 Magic Link 登录。
  - 支持登录、注销、刷新后保持登录和当前用户状态展示。
  - 后续再扩展密码登录、Google/GitHub OAuth。
- 账号定位：
  - 账号是上级身份层，不替代同步码。
  - 同步码是首页空间的访问凭证。
  - 一个账号可以挂载一个或多个首页空间。
  - 每个首页空间先关联一个现有同步码空间，首页内容继续沿用当前密文同步机制。
- 登录后的首页空间管理：
  - 用户登录后可将当前本机绑定的同步码“认领”到账号。
  - 用户可在设置页查看账号下已认领的首页空间。
  - 用户可切换某个首页空间作为当前浏览器激活空间。
  - 空间切换会拉取对应首页内容，必须二次确认，避免静默覆盖当前本地首页。
- 全局设置同步：
  - 账号保存语言、展示风格、默认首页空间等最小偏好。
  - 主题、字体、密度和默认搜索引擎已在 Phase 1.6.6 接入；组件默认配置进入后续扩展。
- 安全边界：
  - 账号资料、全局偏好和首页空间索引使用 Supabase RLS 保护。
  - 用户只能查看和管理自己账号下的首页空间索引。
  - 同步码 secret 不应在账号空间列表中明文长期展示。
- 本阶段不做：
  - Stripe 会员。
  - 团队共享。
  - 公开主页。
  - 多同步码凭证管理，已降级为 P4/未定义。
  - 账号自动同步替代同步码。
- 子任务拆分：
  - Phase 1.5.0：确认账号模型和数据库计划，写清 `profiles`、`account_preferences`、`home_spaces` 与现有 `sync_spaces` 的关系。
  - Phase 1.5.1：实现 Supabase Auth 基础登录，包括邮箱 Magic Link、登出、刷新后保持登录。
  - Phase 1.5.2：配置 Resend 作为 Supabase Auth Custom SMTP，解除内置邮件测试通道低额度对 Magic Link 测试和上线的限制。
  - Phase 1.5.3：实现账号资料和全局设置骨架，先保存语言、默认空间和主题偏好占位。
  - Phase 1.5.4：实现同步码认领，把当前本机绑定的同步码挂到账号下的首页空间。
  - Phase 1.5.5：实现首页空间列表与切换，切换时复用现有同步码拉取流程。
  - Phase 1.5.6：安全回归和线上验证，包括 RLS、跨账号隔离、同步码回归和 GitHub Pages 登录回调。

- 本地存储 key：

  - `homepage:document:v2`：当前本地首页文档。
  - `homepage:sync-code:v1`：当前浏览器保存的同步码绑定。
  - `homepage:reset-backup:v1`：最近一次恢复默认前的本地备份。

  - `homepage:ui-preferences:v1`：当前浏览器的全局 UI 偏好缓存；未登录时作为唯一来源，登录后作为账号偏好镜像。
- 云端最小表：
  - `sync_spaces`: 匿名同步码空间，保存密文首页文档、revision、token hash。
  - `profiles`: 用户基础资料，绑定 Supabase Auth 用户。
  - `account_preferences`: 登录用户全局偏好，保存语言、展示风格、默认首页空间等。
  - `home_spaces`: 账号下的首页空间索引，关联某个 `sync_spaces` 记录。
  - `homepage_documents`: 后续可选表；如果未来不再用同步码密文空间承载账号首页，可迁移为账号首页明文或密文文档表。
- 服务端接口：
  - 当前 GitHub Pages static export 下优先使用 Supabase RPC 和 browser client。
  - `create_sync_space`
  - `pull_sync_space`
  - `push_sync_space`
  - `force_push_sync_space`
  - `revoke_sync_space`
  - `activate_home_space`
  - Phase 1.5 账号资料、偏好和空间索引主要使用 Supabase table API + RLS；默认空间激活使用 RPC 保证原子一致性。

## Security Requirements

- 所有 URL 只允许 `http://` 和 `https://`。
- 页面内容不能用用户输入直接拼 HTML；React 阶段使用文本渲染，避免 XSS。
- 外链统一 `target="_blank"` + `rel="noopener noreferrer"`。
- 同步码必须使用强随机数，不允许用户自定义弱密码作为同步码。
- 匿名同步数据默认加密后上传，服务端不读取明文。
- 登录用户数据必须启用 Supabase RLS。
- 登录用户只能管理自己的 `profiles`、`account_preferences` 和 `home_spaces`。
- 账号空间列表不长期明文展示同步码 secret；同步码 secret 只在创建或用户主动复制时出现。
- Supabase service role、同步 token hash、未来支付密钥只能存在服务端。
- 上传图片必须限制文件类型、大小和尺寸。
- 编辑、同步、登录接口加基础 rate limit。
- 删除账号、清空首页、覆盖云端配置、切换首页空间都必须二次确认。

## Acceptance Criteria

- 未登录用户可完整编辑分组和网站，刷新浏览器后内容仍保留。
- 未登录用户换浏览器时看不到旧配置，符合“本地生效”预期。
- Phase 1.9A 完成设置页、首页空间创建、Banner/背景面板和网站收集区编辑入口的 UI/UX 优化，桌面、触屏和键盘路径均可发现关键操作。
- Phase 1.9B 完成浏览器收藏/标签导入需求集设计，明确普通网页、用户主动导入文件、粘贴 URL 和浏览器扩展之间的权限边界，以及大批量收藏导入的清洗、预览和回滚策略。
- 用户创建同步码后，在另一台设备输入同步码能恢复同一首页。
- 同步码模式下，一台设备编辑后，另一台设备刷新或重新聚焦后能看到更新。
- 登录用户可将当前同步码首页空间认领到账号。
- 登录用户可查看账号下已认领的首页空间列表。
- 登录用户可切换账号下的首页空间；切换前有明确确认，切换后能恢复对应首页。
- 登录用户换设备登录后能看到自己的首页空间索引，并选择某个空间恢复首页。
- 登录用户注销后，本地仍可显示缓存，但不再继续云同步。
- Todo 和月历组件能添加、删除、排序、重命名和折叠，并写入统一 `widgets` 数据结构。
- Banner/背景图在登录用户下可上传、保存和恢复。
- 普通页面 URL 本身不赋予编辑权限。
- 错误、冲突、离线、保存中、已保存状态都有明确 UI 反馈。

## Assumptions

- 0-3 个月目标是可公测 MVP，不追求完整会员支付。
- 未登录和单同步码模式仍按单首页处理；账号阶段开始支持多个首页空间索引。
- Phase 1.5 的多首页能力是“账号管理多个同步码首页空间”，不是完整团队空间系统。
- 同步码不是账号身份系统，而是首页空间的低摩擦访问凭证。
- 同步码泄露意味着对应首页空间编辑权泄露；账号和 RLS 保护的是空间索引、全局偏好和未来会员权益。
- 为空间生成新同步码和多同步码凭证管理已降级为 P4/未定义；近期继续强调账号登录、账号托管和首页空间主路径。
- AI 能力作为 Phase 2 核心主线进入 3 个月之后的阶段；组件市场、团队工作区、Stripe 支付同样进入 Phase 2 或更后阶段。
