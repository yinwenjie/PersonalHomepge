# 同步请求优化备忘

## 背景

当前首页已经支持同步码机制。绑定同步码后，前端会通过 Supabase RPC 自动拉取和自动上传首页数据，保证多浏览器、多设备之间的数据尽快一致。

这个机制在 MVP 阶段可用，但它存在一个后续需要优化的点：自动拉取目前使用固定轮询和焦点触发，会在用户没有任何修改时也持续访问 Supabase。

## 当前请求逻辑

### 自动拉取

绑定同步码后，前端会在以下时机调用 `pull_sync_space`：

- 页面启动并读取到本机同步码绑定后，立即自动拉取一次。
- 浏览器窗口重新获得焦点时尝试拉取。
- 页面从隐藏变为可见时尝试拉取。
- 页面打开期间每 30 秒尝试拉取一次。

自动拉取的保护条件：

- 没有绑定同步码时不请求。
- 当前处于冲突状态时不自动拉取。
- 正在同步中时不重复请求。
- 编辑弹窗打开时不自动拉取。
- 自动拉取有 10 秒 cooldown，避免焦点和可见性事件连续触发。

注意：当前 `pull_sync_space` 会更新 `last_pulled_at`，因此它不是纯读请求；每次自动拉取都会造成一次数据库写操作。

### 自动上传

当前本地文档 `revision` 增加后，前端会在 1800ms debounce 后调用 `push_sync_space`。

典型触发包括：

- 新增、编辑、删除分组或网站。
- 拖拽排序分组。
- 拖拽排序或跨组移动网站。
- 导入 JSON 等通过 `commitHomeDocument()` 保存的操作。

自动上传的保护条件：

- 必须已绑定同步码。
- 当前不是冲突状态。
- 当前没有正在进行的同步请求。
- 本地 `documentValue.revision` 必须大于 `binding.lastSyncedDocumentRevision`。

## 风险

- 用户只要绑定同步码并打开页面，即使没有任何修改，也会周期性请求 Supabase。
- 如果多个浏览器标签页同时打开，每个标签页都会独立轮询。
- `pull_sync_space` 会写入 `last_pulled_at`，所以高频拉取会放大数据库写入量。
- 当用户规模增长后，固定 30 秒轮询可能成为 Supabase 免费额度和数据库写入压力的主要来源。

## 后续优化方向

### Phase A：降低轮询频率

短期可将自动拉取策略调整为：

- 页面启动时拉取一次。
- 浏览器重新获得焦点时拉取一次。
- 固定轮询从 30 秒改为 2-5 分钟。
- 页面隐藏时不轮询。

这个方案改动小，能立即降低大多数无效请求。

### Phase B：增加轻量 revision check

新增一个只返回 `revision` 和 `updated_at` 的 RPC，例如 `check_sync_space_revision`。

流程：

- 自动同步先调用轻量 revision check。
- 只有云端 `revision` 大于本地记录的 `remoteRevision` 时，才调用完整 `pull_sync_space`。
- `check_sync_space_revision` 不更新 `last_pulled_at`，避免把检查行为变成数据库写。

这个方案可以显著减少完整文档密文传输和数据库写入。

### Phase C：减少多标签重复同步

通过浏览器端协调减少同一设备多标签重复请求：

- 使用 `BroadcastChannel` 广播同步结果。
- 使用 `localStorage` 写入短期锁，例如 `homepage:sync-lock:v1`。
- 同一浏览器内只允许一个标签页作为同步 leader。

这个方案适合在用户开始长期打开多个标签页时引入。

### Phase D：评估实时同步

后续可评估 Supabase Realtime 或其他实时通知机制：

- 云端 revision 变化时通知客户端。
- 客户端收到通知后再拉取完整文档。

但实时连接会带来连接数、心跳、免费额度和移动端后台行为问题，不建议在 MVP 阶段优先实现。

## 建议优先级

近期优先做 Phase A 和 Phase B：

1. 把固定 30 秒轮询调低到 2-5 分钟。
2. 新增纯读 revision check RPC。
3. 自动拉取改为先 check，再按需 pull。

这两步成本低，收益明确，不影响现有同步码协议和前端数据结构。
